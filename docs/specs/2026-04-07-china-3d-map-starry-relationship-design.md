# Trace Scope 中国 3D 地图底层与满天星关系层设计

## 1. 文档目标

本设计文档用于明确 Trace Scope 后续在地图方向上的扩展边界：

- 地图作为底层空间基底
- 项目 / 地点 / 媒体 / 轨迹关系以“满天星”视觉结构悬浮在地图之上
- 图片与地点、项目、轨迹之间的联系被映射为可浏览的空间关系

本文档只定义设计方向、数据映射方式、视觉结构和技术路线选择，不直接等同于实现完成。

---

## 2. 核心结论

地图在这个方案里不是主角，而是“底层空间承载层”。

必须坚持：

1. 地图放在最底层
2. 关系网络放在地图之上
3. 图片只在交互时展开，不常驻铺满地图
4. 地图负责空间参照，关系层负责叙事表达

最终体验应更接近：

- 夜空下的中国 3D 地图地形 / 城市空间
- 地点像发光星点
- 项目关系像星座
- 轨迹像细线星链
- 图片像围绕地点展开的星簇

而不应该变成：

- 传统 POI 地图
- 普通 marker + popup 地图
- 满屏图片卡片压住地图
- 以地图控件为中心的工具型页面

---

## 3. 设计原则

### 3.1 地图只做空间基底

地图层的职责：

- 提供中国空间范围
- 提供城市 / 地形 / 建筑体量感
- 提供地点坐标定位能力
- 提供镜头运动与空间导航基础

地图层不负责：

- 承担主要叙事信息
- 承担图片主展示
- 用高密度标注替代关系设计

### 3.2 关系层才是叙事核心

用户应首先感知：

- 这些地点彼此有关
- 这些图片不是孤立资源，而是挂在地点和项目上的
- 轨迹和地点顺序构成叙事路径

### 3.3 图片只能在需要时展开

图片如果长期悬浮在地图上，会立即破坏地图的空间结构和视觉节奏。

因此必须采用：

- 默认只显示地点与关系
- hover / click 后再展开地点相关图片
- 图片以星簇、卫星节点、轻量缩略预览形式出现

### 3.4 风格必须保持克制

这个层不是要做“科技感大屏”，而是做“空间叙事的星图层”。

所以视觉应偏向：

- 深色底
- 细线
- 微光
- 低噪音
- 有节奏的亮点分布

而不是：

- 粗线
- 大图标
- 高密度标签
- 夸张动效

## 4. 当前领域模型与地图映射关系

本项目现有核心链路为：

`Project -> Location -> MediaSet / Route`

在地图关系层中，建议映射如下。

### 4.1 Project

地图角色：

- 关系分组单位
- 星座簇所属单位
- 视觉筛选主维度

表现方式：

- 不直接作为地图点
- 作为一组地点与轨迹的“星座归属”
- 选中某个项目时，该项目相关节点与连线高亮

### 4.2 Location

地图角色：

- 主空间节点
- 主星点
- 图片展开的锚点

表现方式：

- 地图上的发光节点
- 大小和亮度略高于其它二级节点
- 点击后展开地点详情与图片关系

### 4.3 Route

地图角色：

- 地点之间的叙事连线
- 星链

表现方式：

- 细线、半透明、可高亮
- 同一项目中的 `Route` 应形成清晰但不过重的关系链
- 线条应弱于节点亮度，避免喧宾夺主

### 4.4 MediaSet

地图角色：

- 地点的二级关系节点
- 图片容器的关系表达层

表现方式：

- 默认不作为常驻地图主节点
- 在地点被选中时，以环绕节点或小型轨道节点形式出现
- 可用于区分 `spin360` 与 `gallery` 两种媒体组

### 4.5 MediaImage

地图角色：

- 最细粒度的图片资源节点

表现方式：

- 默认不常驻地图
- 仅在交互状态下展开
- 可呈现为：
  - 小型发光点
  - 缩略图节点
  - 由地点向外发散的轻量图片星簇

## 5. 视觉层级设计

建议采用三层结构。

### 5.1 底层：3D 中国地图层

作用：

- 提供空间底板
- 提供地理真实感
- 提供 3D 城市或地形深度

视觉要求：

- 深色、低饱和
- 标签弱化
- 建筑体量和地形轮廓可见，但不压过关系层

### 5.2 中层：满天星关系层

作用：

- 展示地点、项目、轨迹之间的结构

构成元素：

- 主星点：Location
- 星链：Route
- 星座簇：Project
- 次级卫星点：MediaSet

视觉要求：

- 发光细点
- 透明细线
- 微弱呼吸感
- 被选中对象明显高亮，其余对象弱化

### 5.3 上层：信息与图片展开层

作用：

- 展示图片预览
- 显示地点详情
- 显示当前选中项目 / 地点的上下文

构成元素：

- 轻量 tooltip
- 小型 glass 弹层
- 图片缩略星簇
- 当前地点信息卡

视觉要求：

- 信息简洁
- 不抢地图
- 不把首页重新做成传统 dashboard

## 6. “满天星关系”交互模型

### 6.1 默认状态

默认展示：

- 中国 3D 地图底层
- 所有可见地点主星点
- 主项目或当前筛选项目的关系线

默认不展示：

- 全部图片缩略图
- 所有图片节点
- 过多文字标签

### 6.2 hover 地点

当用户 hover 某个地点时：

- 该地点亮度上升
- 相关 `Route` 线段增强
- 周围出现轻量图片关系预告
- 可显示地点名称与项目名

### 6.3 click 地点

当用户 click 某个地点时：

- 地点成为核心星点
- 关联 `MediaSet` 作为二级节点展开
- 相关 `MediaImage` 以小型星簇或缩略图卫星形式展开
- 侧边或浮层出现地点详情

### 6.4 click 项目

当用户选择某个项目时：

- 仅高亮该项目所属地点与轨迹
- 其它项目关系降亮度或隐藏
- 该项目关系在地图上形成更像“星座”的路径网络

### 6.5 图片展开策略

图片层建议采用两级展开：

1. 地点层先展开 `MediaSet`
2. 用户继续聚焦后再展开 `MediaImage`

这样能避免一层就把地图塞满。

## 7. 中国 3D 地图技术路线选择

本项目当前重点是“可落地、免费 / 低成本、适合中国范围”。

### 7.1 方案 A：天地图 + 自定义关系层

定位：

- 优先推荐的中国地图基础方案

优点：

- 中国范围适配自然
- 官方地图服务，适合作为底图
- 免费申请 Key
- 有官方 JS API 与开源库支持

适合本项目的原因：

- 地图可放到底层
- 关系层可以自己单独绘制
- 不会强迫页面变成“地图产品”

风险：

- 星图关系层需要自己实现
- 视觉打磨工作量仍然不低

### 7.2 方案 B：高德 JSAPI 2.0 + 自定义关系层

定位：

- 国内最快可出 3D 地图效果的工程路线

优点：

- Web 3D 地图能力成熟
- 中国城市环境表现较强
- 很适合快速做原型

风险：

- 不是开源路线
- 需要注意坐标体系问题
- 关系层依然需要自定义实现

### 7.3 方案 C：CesiumJS + 中国底图适配

定位：

- 3D 能力最强的路线

优点：

- 适合复杂空间视觉表达
- 后续可扩展空间叙事能力更强

风险：

- 工程量最大
- 中国底图适配和资源组织更复杂
- 对当前原型阶段而言可能过重

### 7.4 当前建议

当前阶段优先建议：

1. 首选：MapLibre GL JS + 天地图 WMTS + 自定义真实投影关系层
2. 备选：高德 JSAPI 2.0 + 自定义星图关系层
3. 暂不优先：CesiumJS

原因：

- 现在最重要的是把“地图底层 + 关系表达”建立起来
- 而不是一开始把所有复杂 3D 能力都引入进来

### 7.5 当前阶段的实现重点：真实投影层

当前代码已经具备 `MapLibre + 天地图` 底图能力，但关系层仍主要停留在页面固定布局阶段。

因此后续第一优先级不应继续追加静态视觉，而应补齐“真实投影层”：

- 地点节点基于真实经纬度投影到当前地图视口
- 轨迹连线始终连接投影后的地点节点
- 媒体簇围绕被选中的地点节点展开，而不是固定在页面角落
- 地图平移、缩放、旋转、俯仰时，关系层与底图同步更新

这一阶段不追求 provider-native 3D 叠层或复杂深度遮挡，而是优先建立：

- 正确的地图实例边界
- 正确的经纬度到屏幕坐标转换
- 正确的 overlay 数据流
- 可扩展到后续 3D 深化的投影架构

建议采用：

- 底图：`MapLibre GL JS` + 天地图 WMTS
- 关系层：覆盖地图容器的 `SVG overlay`
- 交互层：少量绝对定位 DOM 节点与信息面板

## 8. 开源 / 免费途径说明

### 8.1 可优先研究的免费 / 低门槛路线

- 天地图官方服务 + JS API
- 天地图开源库
- 高德官方 JSAPI 免费额度方案
- MapLibre GL JS 作为纯开源引擎层

### 8.2 关于“开源”和“免费”的区分

这里必须明确：

- 开源的是“引擎”时，不等于底图数据也天然免费且可直接用于中国 3D 地图
- 免费的是“地图服务”时，也不等于整个技术栈开源

因此这个项目要区分两件事：

1. 地图底图来源
2. 关系层渲染引擎

在这个设计中，可以接受：

- 地图底图由中国地图服务提供
- 关系层由你自己控制和实现

## 9. 与现有代码结构的映射建议

### 9.1 保持现有领域模型不变

不建议为了地图效果重构核心类型。

仍应保留：

- `Project`
- `Location`
- `MediaSet`
- `MediaImage`
- `RouteEntity`

### 9.2 新增视图层模型

建议新增地图视图层模型，而不是污染领域模型。

例如：

- `MapStarNode`
- `MapRelationshipEdge`
- `MapMediaCluster`
- `MapProjectConstellation`

这些模型只负责：

- 地图表现
- 节点样式
- 关系层绘制
- 交互状态映射

不负责替代领域实体。

### 9.3 组件边界建议

建议将地图扩展拆为：

- `MapBase3DView`
  - 负责中国 3D 地图底层与地图实例生命周期
- `MapProjectionOverlay`
  - 负责承接地图实例投影结果，统一渲染覆盖层容器
- `StarRelationshipLayer`
  - 负责消费投影后的星点和关系线
- `MediaClusterLayer`
  - 负责围绕投影锚点展开媒体簇
- `MapRelationshipPanel`
  - 负责当前选中对象详情

禁止把这些逻辑全部塞进一个大组件。

### 9.4 新增投影视图层模型与数据流

为了避免每个组件直接操作地图实例，建议把“真实投影”收束为独立的视图层能力。

建议新增：

- `projectMapGraph`
  - 纯函数，负责把地图节点和边转换为屏幕投影结果
- `useProjectedMapGraph`
  - hook，负责订阅地图事件、驱动重算、合帧更新
- `ProjectedMapNode`
  - 投影后的节点结果，至少包含 `x`、`y`、`isVisible`
- `ProjectedMapEdge`
  - 投影后的边结果，至少包含 `path`、`isVisible`

推荐数据流：

`useMapRelationshipData`
-> `MapPage`
-> `MapBase3DView` 提供 map 实例
-> `useProjectedMapGraph(map, relationshipData)`
-> `MapProjectionOverlay`
-> `StarRelationshipLayer` / `MediaClusterLayer`

边界要求：

- `MapPage` 不直接调用 `map.project()`
- `StarRelationshipLayer` 不直接读取地图实例
- `MediaClusterLayer` 不自己做地理坐标计算
- 所有投影逻辑都收敛到“纯函数 + hook”这一条链路

## 10. 风险与限制

### 10.1 坐标体系风险

如果接入国内地图服务，必须确认：

- 当前经纬度是否与目标地图服务坐标体系匹配
- 是否需要转换

### 10.2 图片层密度风险

如果图片默认常驻，地图会立刻失控。

必须始终控制：

- 默认图片不常驻
- 展开数量有限
- 有层级展开

### 10.3 视觉噪音风险

这个方向最容易做成：

- 科技大屏
- 发光过度
- 线条过多
- 标签过多

必须克制。

### 10.4 工程复杂度风险

如果过早引入过重的 3D 地图引擎，会拖慢整个项目节奏。

因此当前阶段应优先完成：

- 正确的数据映射
- 正确的层级结构
- 正确的视觉秩序

### 10.5 投影同步风险

如果关系层只是视觉叠加而不是依附真实投影，会出现以下问题：

- 地图移动时节点不跟随
- 连线不能反映真实空间关系
- 俯仰后视觉层与底图脱节
- 后续所有交互都建立在错误坐标系上

因此必须把“真实投影同步”视为当前阶段的架构门槛，而不是后续可选优化。

## 11. 验收标准

当这个方向被认为“设计正确”时，应至少满足以下条件：

### 11.1 地图层

- 中国 3D 地图确实处于底层
- 地图具备空间参考价值
- 地图不会压住关系层

### 11.2 关系层

- 地点能作为主星点出现
- 轨迹能形成细线关系链
- 项目关系能形成星座簇观感
- 地图平移、缩放、旋转、俯仰时，节点与连线持续跟随底图同步

### 11.3 图片层

- 图片不会默认铺满地图
- 点击 / 聚焦地点后，图片关系能被清楚展开
- 图片层不会破坏整体空间阅读顺序

### 11.4 架构层

- 领域模型不被污染
- 地图表达使用视图层模型承接
- 地图底层、关系层、图片层边界清楚
- `MapPage` 不直接承载投影计算
- 投影逻辑集中在纯函数与 hook 内

---

## 12. 推荐后续推进顺序

建议按以下顺序实现：

1. 固定底图 provider 边界，确保 `MapBase3DView` 只负责 map 生命周期
2. 建立地图视图层模型
3. 新增投影纯函数，把 `Location` / `Route` 转换为屏幕空间结果
4. 新增投影 hook，订阅地图相机事件并驱动重算
5. 建立统一的 `MapProjectionOverlay`
6. 在 overlay 中渲染 `Location` 主节点
7. 在 overlay 中渲染 `Route` 关系线
8. 增加项目筛选与高亮
9. 增加地点聚焦后的 `MediaSet` 展开
10. 最后再增加图片缩略节点与高级动效

这样能保证先把空间结构做对，再做视觉精修。
